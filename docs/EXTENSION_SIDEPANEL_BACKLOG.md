# ブラウザ拡張: サイドパネルAIチャット（開発予定）

このドキュメントは、将来実装するブラウザ拡張機能
「サイドパネルAIチャット」のバックログです。
現時点では未実装です。

## 目的
- 現在タブの URL と選択テキストを文脈として AI エージェントへ渡す。
- 「このページ内容を根拠に回答して」のような連続対話をしやすくする。
- 調査中の画面を離れず、右側サイドパネルで問い合わせを完結させる。

## MVP要件（実装対象）
1. サイドパネルUI
- チャット入力欄、送信ボタン、会話履歴表示。
- 履歴は1タブ単位で保持（タブを閉じると破棄）。

2. 文脈取り込み
- 「現在タブURLを添付」トグル。
- 「選択テキストを添付」ボタン（未選択時は無効化）。
- 送信時に `url` / `selected_text` / `user_query` をまとめて送る。

3. エージェント連携
- 既存 Gateway の WebSocket (`/ws`) または HTTP 経由で問い合わせ。
- エラー時は再試行導線と原因表示（認証切れ/接続失敗/タイムアウト）。

4. 連続対話
- 直近N件（例: 10件）の会話をコンテキストとして同送。
- 「このページを根拠に」の固定プロンプト挿入ボタン。

5. 最低限の安全制御
- 送信前にドメイン表示（どのページ情報を送るか明示）。
- オン/オフ可能な「ページ情報送信を許可」設定。

## 技術方針
1. 拡張仕様
- Manifest V3。
- `side_panel` + `storage` + `activeTab` + `scripting` + `tabs` 権限を最小採用。

2. 構成
- `background/service worker`: タブ情報取得・Gateway通信管理。
- `sidepanel`: チャットUI。
- `content script`（必要時のみ注入）: 選択テキスト取得。

3. 認証連携
- Gateway側 OIDC セッション（既存）を前提にする。
- 拡張は原則トークン保持しない（Cookieセッション利用）。
- 未認証時は `/auth/login` を開く導線のみ提供。

4. 通信形式
- 初期はテキスト問い合わせのみ（音声なし）。
- 送信payloadの標準形:
```json
{
  "type": "user_text",
  "text": "ユーザー質問",
  "context": {
    "page_url": "https://example.com/post/1",
    "selected_text": "選択部分",
    "conversation_tail": ["..."]
  }
}
```

## 受け入れ基準（Done）
1. サイドパネルから質問送信し、応答が表示される。
2. URL添付ON時、現在タブURLがコンテキストに含まれる。
3. 選択テキスト添付時、選択文がコンテキストに含まれる。
4. 同一タブでの連続対話時、直前履歴が参照される。
5. 未認証時にログイン導線が表示され、認証後に再送できる。
6. ページ情報送信OFF時、URL/選択テキストを送らない。

## 非目標（MVP外）
- 音声対話の組み込み。
- ページ全体DOMの自動スクレイピング送信。
- 複数Gatewayの同時接続。
- 高度なRAGインデックス構築。

## 想定リスク
- 機密情報を含むページ内容の送信リスク。
- サイト権限の過剰要求による審査・運用コスト増。
- タブ単位履歴と全体履歴の混同による誤文脈。

## 次フェーズ候補
1. 参照元リンク付き回答（どのページ文脈を使ったか明示）。
2. ページ内 CVE/PURL 自動抽出チップ。
3. SBOM照合ショートカット（選択語を直接検索）。

## 追加バックログ（通知ボタン分析）
1. 通知メッセージに「分析する」アクションを追加（Chrome拡張）。
2. ボタン押下時、通知本文/リンク/メッセージIDを分析ソースとしてGatewayへ送信。
3. Gateway側で分析フローを非同期実行し、進捗と結果をサイドパネルに表示。
4. API呼び出しはOIDCログイン済みブラウザセッション前提（`credentials: include`）。
5. バックエンド側で最終認可を実施し、監査ログに `user_sub/email + action + timestamp + request_id + result` を記録。

## 追加バックログ（脆弱性起票ロジック）
1. 対応完了目標の営業日計算に「会社休日カレンダー」を適用する。
2. 年末年始・祝日・指定休業日を考慮した期日算出へ切り替える。
3. テーブル駆動で休日マスタを管理し、毎年の手動改修を不要化する。
4. 既存の社内方針（5営業日/10営業日/3か月）に対して、休日適用後の日付を監査ログへ記録する。

## 追加バックログ（品質管理・リリースゲート）
1. 監査ログ/KPI蓄積を実装する。
2. 保存対象: 仮説JSON, ツール結果, 最終JSON, 差分, incident_id。
3. KPI: 必須欠損率、誤抽出率、手修正率、再生成回数、処理時間。
4. 回帰テストゲートを実装する（実通知セットでデプロイ前自動評価）。
5. しきい値未達時はリリース停止する運用にする。

## 追加バックログ（SIDfm依存の段階的解消）
1. SIDfm通知メール依存を外し、公開脆弱性ソース主体へ移行するアーキテクチャを定義する。
2. 一次ソース収集を実装する（NVD/CVE, GitHub Advisory, CISA KEV, ベンダーアドバイザリ）。
3. 収集データを正規化する共通スキーマを導入する（CVE, CPE, CVSS, published, modified, references）。
4. 自社SBOMとの突合パイプラインをSIDfm非依存入力で実行可能にする。
5. 優先度計算を標準化する（KEV該当, EPSS, CVSS, 公開後日数, 資産重要度）。
6. SIDfm入力と公開ソース入力を並走し、差分検証（検知漏れ/誤検知/遅延）を計測する。
7. 差分検証で合格基準を満たしたら、SIDfmを補助チャネルへ縮退する。
8. 最終的にSIDfm停止可否を判断する運用ゲート（SLO, 監査要件, 障害時ロールバック）を定義する。
