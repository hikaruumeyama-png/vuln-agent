vuln-agent Architecture Flow

SIDfm代替 脆弱性インテリジェンス基盤 — 11ソース自動収集 / AI分析 / SBOM照合 / Chat通知

脆弱性管理の自動化 — なにが変わるのか

有償サービス (SIDfm) への依存を解消し、AIが脆弱性情報を自動で収集・判断・通知する仕組みに置き換えます。

📡

情報ソース

11種
NIST / CISA / JVN / Cisco / Microsoft 他

SIDfm (1社) に頼っていた情報源を、世界の公的DB + ベンダー公式APIに拡大。見逃しリスクを大幅に低減。

🤖

AI自動判断

24時間365日
人手ゼロで収集・照合・通知

30分ごとに自動巡回。自社SBOMと照合し、関係のある脆弱性だけを担当者に通知。ノイズを排除。

既存運用を踏襲

通知も分析もGoogle Chat上で完結。@メンションするだけでAIが影響度・対応期限・担当者を即回答。

自動脆弱性検知パイプライン

Cloud Schedulerが定期的にPollerを起動。各ソースから差分取得 → 重複排除 → Pub/Sub → Intake → SBOM照合 → Chat通知。人手不要の完全自動。

  subgraph B["vuln-feed-poller (Cloud Functions)"]
    B1["アダプター選択\n(11種)"] --> B2["fetch_recent(since)"]
    B2 --> B3["VulnEntry正規化"]
  end

  B3 --> C{"重複チェック\nvuln_dedup"}
  C -->|"NEW (初出)"| D["Pub/Sub\npublish"]
  C -->|"SKIP (既知)"| E["sources_seen\n更新のみ"]

  D --> F

  subgraph F["vuln-intake-worker (Cloud Functions)"]
    F1["VulnEntry\nデコード"] --> F2["SBOM照合\n(PURL / 製品名)"]
    F2 -->|該当あり| F3["担当者特定\n→ Chat通知"]
    F2 -->|該当なし| F4["ログのみ\n(通知しない)"]
  end

  F3 --> G["Google Chat\nカード通知"]
  F3 --> H["BigQuery\n履歴記録"]
  F4 --> H

重複排除: 複数ソースが同じCVEを報告しても初回のみ通知。2回目以降は sources_seen に追記して追跡のみ。
SBOM非該当: 自社に無関係な脆弱性は通知せずログ記録のみ。ノイズを排除。

SBOM照合 → 通知判定フロー

脆弱性の affected_products を SBOM と突合。PURL → 製品名 → ベンダー名の順で照合し、該当があれば担当者を特定して通知カードを送信。

flowchart TB
  V["新規脆弱性\n(VulnEntry)"] --> P1{"PURLあり?"}

  P1 -->|Yes| S1["search_sbom_by_purl()"]
  P1 -->|No| P2{"製品名あり?"}

  P2 -->|Yes| S2["search_sbom_by_product()"]
  P2 -->|No| P3{"ベンダー名あり?"}

  P3 -->|Yes| S3["search_sbom_by_product(vendor)"]
  P3 -->|No| NONE["該当なし"]

  S1 --> MERGE["結果マージ\n(重複除去)"]
  S2 --> MERGE
  S3 --> MERGE

  MERGE --> CHK{"マッチあり?"}

  CHK -->|Yes| OWNER["担当者特定\n(owner_mapping\nパターンマッチ)"]
  CHK -->|No| NONE

  OWNER --> NOTIFY["Chat通知カード"]

  NOTIFY --> C1["CVE-ID + タイトル"]
  NOTIFY --> C2["深刻度 + CVSS"]
  NOTIFY --> C3["影響システム一覧"]
  NOTIFY --> C4["対応期限\n(内部 / 公開リソース別)"]
  NOTIFY --> C5["修正バージョン"]
  NOTIFY --> C6["CVEリンク\nベンダーアドバイザリ"]

  NONE --> LOG["ログのみ記録\n(通知しない)"]

  style V fill:#4c1d95,stroke:#8b5cf6,color:#e0e7ff
  style NOTIFY fill:#065f46,stroke:#10b981,color:#d1fae5
  style NONE fill:#1f2937,stroke:#6b7280,color:#d1d5db

照合優先順位: PURL (パッケージURL) が最も正確。次に製品名+バージョン範囲。最後にベンダー名 (製品名がない場合のフォールバック)。
担当者特定: owner_mapping テーブルで glob パターンマッチ。最も具体的なパターンを優先 (完全一致 > ワイルドカード)。

脆弱性ソース一覧 (11種)

4公開DB + 4ベンダーAPI + 3スクレイピング。APIソースは30分、スクレイピングは6時間間隔でポーリング。

ソース種別間隔 APIキーデータ形式状態
CISA KEV公開DB30分 不要JSON カタログ 稼働中
NVD公開DB30分 登録済みREST API v2.0 稼働中
JVN公開DB30分 不要RSS / Atom 稼働中
OSV公開DB30分 不要REST API 稼働中
Cisco CSAFベンダーAPI30分 登録済みOAuth2 + CSAF 稼働中
MSRCベンダーAPI30分 不要 (任意)CVRF XML 稼働中
FortinetベンダーAPI30分 不要RSS フィード 稼働中
AlmaLinuxベンダーAPI30分 不要Errata JSON 稼働中
Zabbixスクレイピング6時間 -Playwright + Gemini Scraper未デプロイ
MOTEXスクレイピング6時間 -Playwright + Gemini Scraper未デプロイ
SKYSEAスクレイピング6時間 -Playwright + Gemini Scraper未デプロイ
スクレイピング3ソースは vuln-scraper (Cloud Run) のデプロイが必要。Playwright で対象ページを取得し、Gemini Flash で脆弱性情報を構造化抽出する。
